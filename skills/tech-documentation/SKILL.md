# Technical Documentation Guide

개발자가 실제로 읽고 싶어하는 문서를 작성하기 위한 원칙입니다.

## Core Principles

### 1. Why Before What
무엇을 하는지 설명하기 전에 **왜 존재하는지** 먼저 설명합니다.

```
❌ "UserService는 사용자 관련 기능을 제공합니다."

✅ "사용자 로직이 여러 곳에 흩어져 있으면 일관성 유지가 어렵습니다.
   '삭제된 사용자 데이터 처리 방법' 같은 정책은 중앙 관리가 필요합니다.
   UserService가 이 역할을 담당합니다."
```

**패턴**: [해결하려는 문제] → [이 모듈의 접근 방식] → [구체적 기능]

### 2. Abstract Lands in Concrete
모든 추상적 설명 뒤에는 **구체적 예시**가 따라야 합니다.

```
❌ "이 함수는 데이터를 변환합니다."

✅ "이 함수는 데이터를 변환합니다. 예를 들어, 사용자가 CSV를 업로드하면
   각 행을 User 객체로 파싱하고, 유효성을 검사한 뒤,
   데이터베이스 저장에 적합한 형식으로 변환합니다."
```

**사용 표현**: "예를 들어," "구체적으로," "실제로는"

### 3. Smart Reader, No Context
기초는 과하게 설명하지 말고, **프로젝트 맥락**은 철저히 설명합니다.

```
❌ "REST API는 HTTP 프로토콜을 사용하는 아키텍처 스타일로..." (독자 과소평가)

❌ "POST /users로 생성하세요." (맥락 없이 던짐)

✅ "사용자 생성은 POST /users입니다. 주의: 이메일 중복 검사는 동기로 실행되지만,
   환영 이메일은 비동기 큐에 추가됩니다. 이메일 서버 장애가 가입을 막지 않습니다."
```

### 4. Don't Hide Tradeoffs
모든 설계에는 타협이 있습니다. **제한사항과 알려진 이슈**를 명시합니다.

```
❌ "이 아키텍처는 확장성이 뛰어납니다." (장점만)

✅ "이 아키텍처는 수평 확장이 쉽습니다. 대신 트랜잭션 일관성을 포기했습니다.
   결제처럼 강한 일관성이 필요한 부분은 별도 처리가 필요합니다. (결제 모듈 참조)"
```

**포함 항목**: 장점, 의도적으로 포기한 것, 알려진 제한/기술 부채, 엣지 케이스

### 5. Documentation Has Flow
독자를 자연스럽게 **개요에서 상세**로 안내합니다.

**순서**: 한줄 요약 → 필요한 이유 → 빠른 시작 (5분) → 전체 그림 → 상세 → 고급 주제

## Writing Rules

### 문장
- 한 문장 = 한 아이디어
- 능동태 > 수동태
- 짧은 문단 (2-4문장)
- 기술 용어는 첫 등장 시에만 설명

### 구조
- 제목은 내용 요약 ("이 앱이 해결하는 문제" not "개요")
- 리스트는 열거가 정말 필요할 때만
- 코드 예시는 복사-붙여넣기로 실행 가능하게
- 핵심 포인트는 주석으로 표시

### 톤
- 동료에게 설명하듯 자연스럽게
- 정보 공유, 강의 아님
- 자신감 있되 독단적이지 않게

## Document Templates

### Full App Documentation

1. **한줄 요약**
2. **이 앱이 존재하는 이유** (문제, 기존 한계, 접근 방식)
3. **빠른 시작** (요구사항, 설치, 첫 시나리오)
4. **아키텍처 개요** (다이어그램, 컴포넌트, 데이터 흐름, 설계 근거)
5. **핵심 개념** (용어, 도메인 지식, 규칙)
6. **모듈 상세** (책임, 인터페이스, 내부 동작, 의존성, 주의사항)
7. **데이터 모델** (엔티티, 관계, 스키마 의도)
8. **API 레퍼런스** (해당 시)
9. **설정 및 환경**
10. **알려진 제한사항 및 향후 계획**
11. **기여 가이드**

### API Endpoint Documentation

```markdown
## POST /users
사용자 계정을 생성합니다.

### 언제 사용하나요?
회원가입 화면에서 새 사용자를 등록할 때 호출합니다.

### Request Parameters
| Name | Type | Required | Description |
|------|------|----------|-------------|
| email | string | Yes | 사용자 이메일 (고유) |
| password | string | Yes | 비밀번호 (최소 8자) |

### Request Body
```json
{
  "email": "user@example.com",
  "password": "securepassword123"
}
```

### Response (201 Created)
```json
{
  "id": "usr_123",
  "email": "user@example.com",
  "createdAt": "2024-01-15T10:30:00Z"
}
```

### Error Codes
| Code | Description | Resolution |
|------|-------------|------------|
| 409 | 이메일 중복 | 다른 이메일 사용 |
| 422 | 유효성 검사 실패 | 요청 형식 확인 |

### curl Example
```bash
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"email": "user@example.com", "password": "securepassword123"}'
```
```

## Quality Checklist

문서 완성 전 확인 사항:

- [ ] 새 개발자가 15분 내에 전체 그림을 이해할 수 있는가?
- [ ] "왜"가 충분히 설명되어 있는가?
- [ ] 구체적 예시가 있는가?
- [ ] 기술적 내용이 정확한가?
- [ ] 코드 예시가 실제로 동작하는가?
- [ ] 트레이드오프와 제한사항이 명시되어 있는가?
- [ ] 빠른 시작 가이드가 실제로 동작하는가?
- [ ] 자주 필요한 정보에 쉽게 접근할 수 있는가?
- [ ] 용어가 문서 전체에서 일관되는가?

## Style Adjustments

상황에 맞게 조절:

- **간결하게**: 각 섹션 200단어 이하, 핵심만
- **깊이 있게**: 시니어 아키텍트도 인사이트를 얻도록
- **초보자용**: 경력 1년 미만 개발자도 이해하도록
- **레퍼런스용**: 검색 가능한 빠른 참조 구조
- **의사결정자용**: 근거와 고려한 대안에 집중
