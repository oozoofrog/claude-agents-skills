# TDD와 작은 리팩토링 원칙

이 스킬은 TDD와 작은 리팩토링을 중심으로 한 소프트웨어 개발 원칙을 제공합니다.

## 언제 적용되는가

- 코드 리뷰 시
- 리팩토링 요청 시
- 새 기능 구현 시
- 기술 부채 논의 시
- 테스트 작성 시
- 코드 품질 개선 시

## 핵심 철학

### 1. 작동하는 코드 먼저, 그 다음 좋은 코드
먼저 작동하게 만들고, 그 다음 올바르게 만들며, 필요할 때 최적화한다.

### 2. 작은 단위 변화
한 번에 한 가지만 변경. 각 변경 후 테스트 통과 확인.

### 3. 테스트가 설계를 이끈다
테스트하기 어려운 코드는 설계가 잘못된 신호.

### 4. 명확한 의도
코드는 작성자의 의도를 명확하게 드러내야 함.

## 빠른 참조: Red-Green-Refactor

```
1. RED: 실패하는 테스트 작성
2. GREEN: 테스트를 통과하는 최소한의 코드 작성
3. REFACTOR: 중복 제거, 명확성 개선 (테스트는 계속 통과)
```

## 리팩토링 시 필수 질문

1. 이 변경이 테스트를 깨뜨리는가?
2. 한 번에 한 가지만 변경하고 있는가?
3. 변경 의도가 코드에 명확히 드러나는가?
4. 더 단순하게 만들 수 있는가?
